一、架构层：最大的结构性问题

1. 主循环应重构为有限状态机（FSM）

当前 run_automation() 是一个超过 200 行的巨型 while 循环，内含 5~6 层嵌套 if-else。这在本质上就是一个没有被显式声明的状态机——你有"交易行主界面"、"商品详情页"、"购买确认中"、"购买成功弹窗"、"售空"、"异常弹窗"、"未知页面"等至少 7 种状态，但全部用 if-else 扁平地铺在一起，导致流程极难追踪，新增场景时也容易引入逻辑冲突。

建议用枚举 + 字典分发来重构：

python

class State(Enum):    SCANNING = auto()       # 交易行扫描价格中    BUYING = auto()         # 点击购买+确认中    WAIT_RESULT = auto()    # 等待购买结果    SUCCESS = auto()        # 购买成功处理    FAILED = auto()         # 购买失败处理    SOLD_OUT = auto()       # 商品售空    PRICE_MISMATCH = auto() # 价格不符退出    EMPTY_SHOP = auto()     # 空置店铺    ABNORMAL = auto()       # 异常场景恢复    PAUSED = auto()         # 暂停挂机# 每个状态对应一个纯函数，输入当前帧，输出下一个状态STATE_HANDLERS = {    State.SCANNING: handle_scanning,    State.BUYING: handle_buying,    # ...}current_state = State.SCANNINGwhile True:    handler = STATE_HANDLERS[current_state]    current_state = handler(frame, context)

运行



这样做的核心收益是：每个状态的进入条件、退出条件、超时兜底都在一个 独立的、可单独测试的函数 里闭合，不会互相污染。

二、线程安全：当前存在竞态条件

这是整个脚本中最危险的隐患。



问题描述

keyboard.add_hotkey('f12', toggle_pause) 会在一个独立的后台线程中触发 toggle_pause()，而 toggle_pause() 内部会读写 IS_PAUSED、total_running_time、last_resume_time。与此同时，主循环也在不断读写这些变量。Python 的 GIL 只能保证单条字节码的原子性，不能保证 toggle_pause() 中多行赋值的原子性。

极端情况下，主循环在 if not IS_PAUSED and last_resume_time is not None 这一行读到 IS_PAUSED=False，但紧接着 toggle_pause() 将 last_resume_time 设为 None，主循环下一行 time.time() - last_resume_time 就会抛出 TypeError。虽然你外面有 except 兜着，但运行时间的累计值可能已经被污染了。



修复方案

python

查看全部

import threading_state_lock = threading.Lock()def toggle_pause(ctx: BotContext):    with _state_lock:        ctx.is_paused = not ctx.is_paused        if ctx.is_paused:            if ctx.last_resume_time is not None:                ctx.total_running_time += (time.time() - ctx.last_resume_time)                ctx.last_resume_time = None        else:            ctx.last_resume_time = time.time()

运行



主循环中所有读取 is_paused / last_resume_time 的地方也需要加锁或者通过原子快照读取。

同理，current_balance、success_count、fail_count 等在 overlay_root.after() 回调（Tkinter 主线程）和主循环线程之间共享，也需要加锁保护。

三、性能层：进一步压榨的空间

1. 模板匹配可以预计算 + 降级加速

当前 get_number() 中对 0-9 共 10 个模板逐一执行 cv2.matchTemplate()，每帧最多做 10 次卷积。在你的像素预检通过的前提下，这已经是合理范围，但仍有两个优化点：

第一，预存模板尺寸，避免 OpenCV 每次内部重新获取。第二，考虑将 10 个模板合并为一张拼接图，用单次匹配 + 位置偏移反推数字。这样卷积次数从 10 降为 1，理论加速约 5~7 倍。实现较复杂，但在帧率竞争极限场景下值得做。



2. gc.disable() 的风险

你在启动时执行了 gc.disable()，然后在空店循环中手动 gc.collect()。这意味着在其他所有路径中（购买成功/失败/售空/异常恢复），循环引用产生的垃圾永远不会被回收。DXCam 的帧对象和 NumPy 数组如果存在交叉引用，内存会持续增长。

我建议改为调高垃圾回收阈值而不是完全关闭：

python



gc.set_threshold(5000, 50, 20)  # 大幅降低回收频率但不完全禁用

运行



这样既避免了 GC 在关键抢购路径上突然触发导致的几十毫秒卡顿，又不会让内存无限制增长。



3. OCR 不需要每次都跑完整管线

get_balance() 中的 resize → sharpen → pad → OCR 管线在每次被调用时都全量执行。但余额区域的内容通常在两次购买之间不会变化。可以加一个简易变化检测：

4. precise_sleep 末段的忙等待

python



while time.perf_counter() < target_time:    pass  # 这里是 100% CPU 占用的自旋锁

运行



在你需要微秒级精度的场景（购买→确认的 10ms 间隔）这是必要的。但在 PRE_EXIT_CLICK_DELAY = 0.03 这样 30ms 的场景下，建议把忙等待的阈值从 3ms 提高到例如 1ms，减少无意义的 CPU 空转：

python



def precise_sleep(seconds, spin_threshold=0.002):    if seconds <= 0: return    target = time.perf_counter() + seconds    while target - time.perf_counter() > spin_threshold:        time.sleep(0.001)    while time.perf_counter() < target:        pass

运行



对于不需要极限精度的调用（退出点击前的 30ms），可以传入更大的 spin_threshold。

四、健壮性层

1. 裸 except 必须消灭

整个脚本中有大量 except: pass 或 except: return False。这会吞掉所有异常，包括 KeyboardInterrupt、SystemExit、MemoryError 等致命错误。最低限度应改为 except Exception，核心路径建议捕获具体异常并记录日志：

python



except (cv2.error, ValueError) as e:    ui_print(f"⚠️ 识别异常: {e}", save_log=True)    return None

运行

五、代码质量层（小项汇总）

日志系统：建议用 Python 标准库 logging 模块替代自写的 ui_print，可以自动处理文件轮转、日志级别过滤、格式化，减少手写 I/O 的 bug 风险。

PushPlus Token 硬编码：敏感凭证应通过环境变量或配置文件注入，不应出现在源码中。一旦代码泄露，Token 会被滥用。

进程亲和性掩码 0x0055：这将进程绑定到 CPU 核心 0、2、4、6（假设 8 核），注释中没有解释为什么选择这几个核心。如果目标机器核心数不同，这个掩码可能适得其反。建议加注释说明意图，或改为自动计算。

pyautogui.FAILSAFE = False：这关闭了 pyautogui 的鼠标角落安全退出机制。在你已经用 Win32 API 做点击的前提下，pyautogui 基本只在初始化时被引用，这行其实可以连同 import pyautogui 一起移除，减少一个不必要的依赖。

CONFIRM_DELAY = 0.01：购买到确认之间只有 10ms，这在网络延迟较大时确认弹窗可能还没渲染出来就被点了。如果游戏的确认按钮是固定位置且点击不到也无副作用，那没问题；但如果点击位置在弹窗出现前对应的是其他可交互元素，可能会产生误操作。值得加一行注释说明为什么 10ms 是安全的。

总结优先级排序

如果要按投入产出比排序，我建议的改造优先级是：

第一优先级（安全/正确性）——线程锁保护共享变量、消灭裸 except、DXCam 旧帧时间戳保护。这三项不改迟早会出现难以复现的诡异 bug。

第二优先级（可维护性）——状态机重构、全局变量收敛为上下文对象、配置外置。这决定了代码后续能不能持续迭代而不崩盘。

第三优先级（性能）——OCR 变化检测跳过、模板合并单次匹配、GC 阈值调优。这些是在已有基础上的锦上添花。
